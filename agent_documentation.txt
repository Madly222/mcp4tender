TENDERENGINE — MAINTENANCE MAP (for AI handoff)
================================================
B2G tender-monitoring platform for SC Rapid Link SRL (Moldova). Python 3.10, FastAPI + SQLite,
Ubuntu 22.04, venv at ~/tenderengine/.venv, no Docker. User: Victor (speaks Russian; UI is English).

HOW TO USE THIS FILE
--------------------
The project is split into small, focused files grouped by COMPONENT below. To work on ONE thing in a
fresh chat, send only that component's "SEND" files (plus this map) — OR, if the assistant has git
access in its sandbox, tell it to clone the repo and read the files itself (see GIT ACCESS below).

GIT ACCESS
----------
Repo: https://github.com/Madly222/mcp4tender (public). The assistant's sandbox has github.com,
codeload.github.com and raw.githubusercontent.com allowlisted, so it can run
`git clone --depth 1 https://github.com/Madly222/mcp4tender.git` directly instead of waiting for a
tar upload. IMPORTANT: the assistant cannot push (no credentials) — Victor applies changes locally
(tar or a pasted diff) and pushes himself. Always ask "did you push?" before assuming the repo the
assistant sees is current; a fresh clone at the start of a new chat is the reliable way to check.

THIS FILE IS PUBLIC
-------------------
agent_documentation.txt is committed to the PUBLIC GitHub repo above. Never write into it: public IPs,
hostnames, ports reachable from the internet, network topology, logins, tokens, API keys, or anything that
tells a stranger where the live instance is and how it is exposed. Deployment specifics live outside git
(ask Victor). Describe mechanisms ("the app binds web.host/web.port; a gateway NATs an external port to it"),
never addresses.

CODE STYLE (STRICT)
-------------------
- NO COMMENTS anywhere in code.
- Small modules, one concern each. If a file grows past ~300 lines, split it.
- Config-driven: no hardcoded values that belong in ConfigStore.
- Delivery: for a handful of changed files, ship a tar (~tenderengine_<name>.tar.gz) with install command
  `cd ~/tenderengine && tar xzf ~/tenderengine_<name>.tar.gz -C . && systemctl --user restart tenderengine`.
  For a single small/doc file, a pasted diff or the raw file to overwrite is fine too. Either way, Victor
  commits + pushes after applying — the assistant cannot do this itself.

RUN TESTS
---------
python -m pytest tests/ -q --ignore=tests/test_phase4b.py
(test_phase4b needs pymupdf; passes on the server, skip in a bare sandbox.)
Web/restart tests monkeypatch web.context._schedule_restart.

SANDBOX NOTES (for the assistant)
---------------------------------
Filesystem resets between turns. Prefer `git clone --depth 1 https://github.com/Madly222/mcp4tender.git`
at the start of a session over waiting for an upload — it's the current source of truth (assuming Victor
pushed). If no network/git access in a given session, fall back to rebuilding from delivered archives.
Network allowlist has pypi.org, github.com, codeload.github.com, raw.githubusercontent.com, but NOT
arbitrary sites or the Playwright/Chromium CDN, so live crawling of real target sites (tenderhub.md etc.)
cannot be tested end-to-end in the sandbox — mock preview_site/analyze_site/detect_site_profile instead.
Chromium IS available for local file:// render tests and screenshots. fastapi + python-multipart are NOT
preinstalled — `pip install fastapi python-multipart --break-system-packages` if testing route functions
directly (calling them as plain Python functions works fine without a real ASGI server/TestClient; just
pass every Form(...) parameter explicitly since the FastAPI default sentinel isn't resolved that way).

=====================================================================================================
GLOBAL CONTRACTS (shared by everything)
=====================================================================================================
- Entry points: app.py bootstrap()->(conn,store,seeded). web/server.py create_app(db_path)->FastAPI.
- ConfigStore (engine/config_store.py): store.get(key,default) / store.all() / store.set(key,value,actor,note)
  / store.reload() / store.seed_defaults(dir). store.conn is the sqlite connection. Keys are namespaced
  strings; values may be dicts or scalars (e.g. "sources.genericweb" -> dict, "collect.max_age_days" -> int).
- Defaults live in config/defaults/*.json (seeded once at bootstrap; new keys auto-added, existing untouched).
- Pipeline status flow (tenders.status): collect->"new"; triage->"triaged"; extract->"extracted";
  applicability->"analyzed"; suppliers->"sourced".
- tenders.origin (TEXT, auto-migrated): how a tender entered — "incremental" (daily/scheduled front-page
  check) vs "backfill" (manual history crawl from Sites). Set by engine/collectors._store_item at insert.
  This is the INPUT to segmentation (New vs General/History), separate from status. See [16].
- SEGMENTS (workflows/segments.py, a DERIVED view, not a stored column): every tender is classified live
  into new | history | archive from origin + created_at + pub/deadline dates. New = incremental finds
  inside results.new_days; archive = deadline passed OR aged past results.archive_after_days; else history.
- external_id: genericweb = "<site_id>:<sha1(url|ref|title)[:16]>"; mtender = ocid-based.
- DB tables (engine/db.py, CREATE IF NOT EXISTS + _migrate/_column_exists on each boot): tenders, verdicts,
  extractions, verifications, suppliers, raw_documents, source_state, source_runs, pipeline_runs, stage_runs,
  llm_cache, configs, config_meta, audit_log, crawl_state, dismissed_tenders, supervisor_runs.
- LLM: engine/llm.py LLMGateway(store,conn).complete(stage,system,messages,max_tokens,prefill) with llm_cache.

    SEND: engine/config_store.py, the relevant config/defaults/*.json, web/config_meta.py

[3] DB / SCHEMA
    Purpose: schema + migrations. Add a table/column here; it auto-applies on next boot.
    Files: engine/db.py
    Key tables for collection: crawl_state(site_id PK, next_url, auth_json, last_run_at, total_collected,
    exhausted, total_estimate, note, detected_count); dismissed_tenders(external_id PK, source, reason,

    SEND: engine/db.py

[4] COLLECTOR FRAMEWORK + AGE FILTER (filtration)
    Purpose: registry, run_collector loop, storage/dedup, and the max-age filter (applies to ALL sources).
    Files: engine/collectors.py
    Contracts: register_collector(name); get_collector(name); CollectContext(config,cursor,source_config,params);
    CollectResult(items,cursor,fetched); CollectedItem(external_id,normalized,raw); Collector.collect(ctx).
    run_collector(source,store,conn,params)->{status,fetched,new,too_old,cursor}. _store_item(conn,source,
    item,origin="incremental") suppresses dismissed_tenders, upserts tenders/raw_documents, and STAMPS
    tenders.origin. _origin_for(params) maps mode->origin: params{mode:"backfill"}->"backfill" (Sites history
    crawl), everything else ->"incremental" (daily/scheduled). This origin drives the New segment.
    AGE FILTER: config key collect.max_age_days (int days, 0=off). _parse_date(str)->ts (ISO YYYY-MM-DD,
    DD.MM.YYYY, DD/MM/YYYY). _tender_ref_ts(normalized)=max of published/deadline/enquiry_deadline. Items older
    than now-days are skipped BEFORE storage (undatable items are KEPT); counted as too_old.
    SEND: engine/collectors.py (+ config/defaults/30_sources.json for the default)

[5] GENERICWEB COLLECTOR — the site "search"/crawler
    Purpose: crawl an arbitrary listing site -> tenders, with pagination, cursor, percent-step, estimate.
    Files: workflows/collectors/genericweb.py
    Site entry (config sites.tenders[]): {id,label,url(base),enabled,step_percent(1-100),render(bool),
    engine(optional str 'builtin'|'crawl4ai' — per-site override of sources.genericweb.engine; unset =
    falls back to the global engine, unchanged legacy behavior)}.
    Source config sources.genericweb: {enabled,timeout,max_input_chars,max_output_tokens,max_pages_per_run,
    incremental_max_pages,fallback_batch,engine('builtin'|'crawl4ai'),wait_ms}.
    Modes via params{mode,site_id}: "backfill" (resume crawl_state cursor, stop at target NEW, advance cursor,
    honest stall note if it stops below estimate), "incremental" ("Check new" front pages), "estimate"
    (page 1 only -> total_estimate + note, stores nothing).
    Key fns: _page_text(url,sc,headers,render,max_chars)->(text,rendered) = THE single fetch entry (routes to
    crawl4ai or builtin); _fetch (builtin get_text/get_text_rendered); _html_to_text; _md_links_to_inline;
    _extract(gw,store,text,url,sc)->(tenders,next_url,estimate) [estimate = total_estimate OR last_page*perpage];
    _increment_page (numeric ?page=N pagination, preferred); _load_state/_save_state/_save_estimate/_save_note;
    _normalize_item (sets normalized 'published'=extracted date, 'deadline', documents, etc.);
    _site_sc(sc,site) -> per-site sc copy with sc['engine'] overridden if site.get('engine') is set (does
    NOT mutate the shared sc dict); used in collect() for both the estimate-mode and main fetch loop.
    SEND: workflows/collectors/genericweb.py (+ engine/collectors.py if touching run/store, + config 140_sites.json)

[6] SITE ANALYZER (preview + profile a site before extracting; auto-detect funnel)
    Purpose: "test" a URL (what tenders are found), "analyze" a site (page_type, has_tenders, follow-links,
    needs_login, recommendation), and auto-pick the cheapest working fetch method for a new site.
    Files: workflows/collectors/genericweb_analyze.py (imports genericweb as _gw; local var gw = LLMGateway).
    Contracts: preview_site(store,conn,url,render=False,auth=None,engine=None)->{chars,rendered,count,titles,
    next,estimate,note}; analyze_site(store,conn,url,render=False,auth=None,engine=None)->{page_type,
    has_tenders,tender_count,follow:[{label,url}],needs_login,recommendation}. `engine` param (added on top
    of the original render/auth) forces sc['engine'] for that one call only, without touching the global
    sources.genericweb config — used by the detect funnel to test methods in isolation.
    detect_site_profile(store,conn,url,auth=None) -> {url,engine,render,count,estimate,needs_login,steps:[...]}:
    runs DETECT_FUNNEL = (builtin/no-render, builtin/render=True, crawl4ai) cheapest-first on ONE test page
    via preview_site; escalates to the next step only if count >= previous_best * DETECT_IMPROVEMENT_RATIO
    (1.2, i.e. needs >=20% more tenders found — tune this constant against real sites if it proves too
    strict/loose); stops immediately once count reaches the parsed estimate; if a step has count==0 and
    chars<400 it runs analyze_site once to check needs_login and short-circuits the funnel if true (no point
    trying pricier engines against a login wall); crawl4ai step is skipped (recorded with "skipped" key, not
    silently dropped) if crawl4ai_fetch.crawl4ai_available() is False. `steps` in the return carries the full
    trace of every attempt for UI display. ANALYZE_PROMPT unchanged. NOTE: reference genericweb helpers via
    _gw.* so monkeypatch/runtime swaps work; same pattern used internally for _save_estimate/_save_note calls
    from web/routes_sites_diag.py's /sites/detect route.
    SEND: workflows/collectors/genericweb_analyze.py (+ genericweb.py for context)

[7] CRAWL4AI ENGINE (swappable collection engine)

    Files: engine/crawl4ai_fetch.py
    Contracts: crawl4ai_available(); fetch_markdown(url,timeout,wait_ms,headers)->markdown. _run_async is a sync

    Install: .venv/bin/pip install crawl4ai ; playwright install chromium.
    SEND: engine/crawl4ai_fetch.py (+ genericweb.py where _page_text picks the engine)

[8] JS RENDER (builtin Playwright)

    Files: engine/render.py
    Contracts: render_available(); get_text_rendered(url,timeout,headers,wait_ms)->html (chromium --no-sandbox).
    SEND: engine/render.py

[9] MTENDER API COLLECTOR (official OCDS source)
    Purpose: reliable full MTender via public.mtender.gov.md OCDS API (mtender.md web is JS -> not scraped).
    Files: workflows/collectors/mtender.py
    Config sources.mtender: {enabled,list_url,record_url_template,portal_url_template,timeout,page_limit,
    max_records_per_run,backfill_days}. normalize_record sets: title, description, buyer, value_amount/currency,
    cpv (list of {id,scheme,description}), main_category, procurement_method, documents, and dates
    'deadline'(=tenderPeriod.endDate = data depunerii), 'enquiry_deadline', 'publication_date'(=datePublished or
    release date), 'date'. DEDUP: base_ocid(ocid) strips stage suffixes (ocds-<p>-<CC>-<digits>); the collector
    keys external_id on the base so planning/tender/award stage-copies of one process collapse to one tender;
    analysis.dedupe_mtender(conn) merges already-stored duplicates (button "Dedupe MTender" on /sites).
    DOCUMENT DEDUP: MTender re-uploads the same attachment on each stage/amendment with a NEW uuid+url (the url
    tail -<epoch_ms> is the version marker), so _documents() would otherwise list the same file many times.
    dedupe_documents(docs) (mtender.py) groups by normalized title (_doc_title_key, falls back to the url
    filename minus the -<ts> tail) and keeps the NEWEST version (_doc_version = the trailing epoch-ms, else
    datePublished), preserving first-seen order. _documents() runs it automatically on every collect. For
    already-stored tenders: analysis.dedupe_documents_db(conn) rewrites each tender's normalized_json.documents
    — exposed as CLI `app.py dedupe-docs [--source mtender]` and the "Dedupe documents" button on /sites.
    SEND: workflows/collectors/mtender.py (+ config/defaults/30_sources.json)

[10] ANALYSIS PIPELINE (triage -> extract -> applicability -> suppliers -> verify)
    Purpose: turn collected "new" tenders into analyzed/sourced ones. Status chain:
    new -> triaged -> extracted -> analyzed -> sourced (failed = a stage errored).
    Files: workflows/triage.py, workflows/extract.py, workflows/applicability.py, workflows/suppliers.py,
    workflows/verify.py ; engine/orchestrator.py (process_stored_tenders(...,limit); process_selected_tenders).
    On-demand runner + funnel: workflows/analysis.py.
    STEP TRACES: each stage's produce_* builds a `steps` list persisted via workflows/trace.py:log_steps(conn,
    tender_id,stage,steps) into the stage_events table (deleted+rewritten per run). Extract/applicability steps
    include the FULL model I/O: "Model input · system prompt", "Model input · content sent", "Model output ·
    raw response" (truncated ~3k/6k/8k). Rendered on the stage tabs (see [13]).
    TRIAGE RULES (workflows/triage.py): _text() scores TITLE + DESCRIPTION only (buyer name EXCLUDED — a cyber/
    police buyer must not make an AC-repair tender relevant). bucket_for(score,config,has_keyword): a tender is
    "relevant" only if it reaches the threshold AND has >=1 positive KEYWORD match; CPV/value alone caps at "gray"
    (CPV prefixes are broad). Keyword/CPV/value weights live in triage.* config (edit via User Settings).
    EXTRACT (workflows/extract.py): no documents -> _fields_from_metadata (obiect from title+CPV, valoare from
    value) with method="metadata"; also pulls data_depunerii (submission deadline stated in docs) as the
    AI-estimated deadline. source_text capped at extract.max_input_chars (200k) to avoid 400 "prompt too long".
    _store_extraction always stores a dict ({"raw":...} for non-dict) so the detail page never crashes.
    APPLICABILITY: _store_verdict never writes NULL (unknown fallback). analysis.py helpers:
    funnel_counts; run_triage/run_extract/run_applicability/run_suppliers/run_all (all take limit=analyze.
    batch_size, default 50, set on the Analyze page, AND scope=one of SCOPES). SCOPES=("all","not_new",
    "new","history","archive"): _scope_ids(conn,store,scope) restricts a run to a segment so a tester can
    re-analyze old data ("not_new"/"history") without touching fresh daily finds ("new"). clear_irrelevant
    (also scope-aware); wipe_collected; reset_analysis;
    requeue_failed; dedupe_mtender(conn) (merge MTender stage-copies by base ocid); clear_stage(conn,stage)
    (per-stage CASCADE clear: deletes that stage + all downstream stage data and rolls tender status back one
    step; wired to the ✕ clear buttons under each stage on /analyze).
    SEND: the specific stage file(s) + workflows/analysis.py + workflows/trace.py + engine/orchestrator.py

[10b] LLM GATEWAY (engine/llm.py)
    complete(stage,system,messages,max_tokens,prefill): picks model from llm.models[stage]; guards empty content;
    on a model-related API error, auto-retries once with llm.models["default"] (so a stale model string like a
    retired sonnet won't break a whole stage). Current default models live in config/defaults/60_llm.json
    (haiku-4-5 default/extract/suppliers; sonnet-5 applicability). If a stage 400s, check the model string.
    Provider selection: select_provider(store).name -> "anthropic" | "stub". The API key is read via
    engine/secrets.get_api_key (env var first, then ~/tenderengine/.env), NOT from config. Health/diagnostics
    live in engine/health.py, surfaced on User Settings — see [16].

[11] WEB — CORE
    Purpose: FastAPI app, auth/context, shared HTML render, styles.
    Files: web/server.py (create_app; includes routers digest/config/sites/analyze/settings/stage/results;
    /healthz /login /logout); web/context.py (request.state.conn/store/readonly/account);
    web/render.py (_e,_ts,_table,_layout,_login,_loose,_vclass,_help,NAV,source_url,...);
    web/styles.py (CSS dark theme).
    AUTH — TWO MODES, decided at request time by _accounts_on(conn) in context.py:
      * COMPANY ACCOUNTS (preferred, active as soon as >=1 row in `accounts`): login+password form ->
        engine/accounts.authenticate -> session row -> cookie te_session (httponly, samesite=lax,
        secure=web.cookie_secure, lifetime web.session_days default 30). An account is a COMPANY credential
        (login, password_hash, company, active), deliberately shareable by several people at that company —
        it is not a per-person user. Brute force is throttled per client IP (login_attempts table;
        accounts.MAX_FAILS=10 within LOCK_WINDOW=900s -> locked, right password included). Manage with
        `app.py user add|list|passwd|disable|enable|delete --login X [--company Y]` (password is prompted
        when --password is omitted, so it stays out of shell history). Password change / disable / delete all
        kill that account's live sessions.
      * TOKEN (legacy fallback, ONLY while zero accounts exist, so an existing install is never locked out):
        _expected_token(store) = env TENDERENGINE_WEB_TOKEN FIRST, then config web.token. GOTCHA THAT BIT US:
        the env var WINS, so setting web.token in config while the env var is set changes nothing — the login
        form keeps checking the env value. Token mode also accepts ?token= in the URL (leaks into logs and
        browser history). Creating the first account turns this whole path OFF, which is the point.
    Passwords: PBKDF2-SHA256, 240k iterations, per-password random salt, hmac.compare_digest (engine/accounts).
    Stdlib only — do not add bcrypt/passlib deps.
    NAV (render.py) — current tabs: Digest, Results, Tenders, Triage, Extract, Applicability, Suppliers,
    Analyze, Sites, Schedule, User Settings, Config. REMOVED (were dev noise / folded into per-tender stage
    logs): Verifications, Coverage, Runs, Stages, Audit.
    Router wiring (server.py include_router): digest, config, sites, analyze, settings, stage, results,
    schedule. Add a new page = new routes_*.py with router=APIRouter() + one include_router line here.
    source_url(source, external_id, portal): builds the MTender portal link from an ocds- external_id
    (default portal template sources.mtender.portal_url_template = https://mtender.gov.md/tenders/{ocid}).
    SEND: web/server.py, web/context.py, web/render.py, web/styles.py (only those you touch)

[13] WEB — OTHER PAGES

      cockpit (sections built from engine/health.py + engine/secrets.py): _api_section (paste/replace
      ANTHROPIC_API_KEY -> secrets.set_api_key writes ~/.env chmod 600; POST /user-settings/apikey; a "test

[18c] MULTI-COMPANY: SOLVED BY DEPLOYMENT, NOT BY SCHEMA (decided Jul 2026, re-confirmed after
    Victor asked for per-company sources/schedule/models/pricing too). If EVERY setting is
    per-company then companies share nothing but the code - and storage that shares nothing is a
    separate database. So: ONE INSTANCE PER COMPANY. INSTANCE IS NOT A VM. Everything runs on ONE
    machine dedicated to the project; an instance is a data directory + a systemd unit + a port.
    ONE code checkout serves all of them, so `git pull` still updates everyone at once:
        ~/tenderengine/          the only checkout, the only .venv
        ~/instances/<company>/   data/tenderengine.db + .env   (one dir per company)
    THE CODE ALREADY SUPPORTS THIS - verified, no change needed:
      * DB_PATH        <- env TENDERENGINE_DB          (app.py:18)
      * .env location  <- env TENDERENGINE_HOME        (engine/secrets.py env_path)
      * port           <- store.get("web.port")        (app.py cmd_web) and the store is read from
                          THAT instance's db, so the port is per-instance for free
    A unit therefore only needs TENDERENGINE_HOME, TENDERENGINE_DB and EnvironmentFile pointed at
    the instance dir, with WorkingDirectory/ExecStart still on the shared checkout.
    Why this and not a multi-tenant core, in case a future agent wants to "improve" it:
      * The engine's real work is a BACKGROUND thread (app.py _start_scheduler_thread), which owns
        its own ConfigStore loaded once at startup and fires run_pipeline() on a 30s timer. Nobody
        is logged in at 03:00. "Swap the config for whoever is currently working" has nothing to
        swap in the main path, and swapping ambient globals is the exact source of cross-company
        leaks. Web requests are already isolated (context.py builds a fresh conn+ConfigStore per
        request) - that isolation is not the problem; the scheduler is.
      * Separate processes give per-company scan times and true parallelism for free: separate
        scheduler threads, separate SQLite files, no lock contention on one db, no GIL sharing.
      * Isolation is physical. A multi-tenant core leaks the moment one WHERE company_id is
        forgotten - and applicability answers "can WE do this", so a leak is not cosmetic.
      * Each company's API key stays in its own .env instead of plaintext in your SQLite.
    Therefore: do NOT add company_id/account_id to tenders, verdicts, suppliers or extractions.
    Do NOT build a config overlay for capabilities.profile / triage.keyword_weights /
    suppliers.catalog / sources / schedule. The "3b multi-tenancy" plan is CANCELLED.

[18d] INBOX = "CAN WE STILL SUBMIT A BID?" - THE ONLY QUESTION THAT MATTERS:
    Victor's rule, and it is source-agnostic on purpose: a tender past the bid-submission window
    is useless to us however well it fits. Portal wording and language are irrelevant - MTender
    says "Depunerea ofertelor" / "Calificarea ofertantilor" / "Anulat", genericweb sites say
    nothing at all. Never build the filter around one portal's vocabulary.
    web/user/lifecycle.state_of(nj, deadline_raw, closed, now) decides:
      1. status OR status_details in results.closed_statuses  -> CLOSED
      2. else an authoritative submission deadline is known    -> CLOSED/OPEN by that date
      3. else                                                  -> UNKNOWN (shown, chip warns)
    THE BUG THIS REPLACED, do not bring it back: state_of used to end with
    `return OPEN if status else UNKNOWN`, treating "a status exists" as proof bidding was open.
    OCDS keeps tender.status = "active" through qualification and award, so every
    qualification-stage tender with no deadline in its record sailed into the inbox. The ONLY
    positive evidence of an open window is a submission deadline still ahead. A status merely
    being present proves nothing.
    Note step 2 uses nj["deadline"] ONLY, never cards.deadline_of's fallback to the date an LLM
    read out of the spec (ex_fields.data_depunerii). A date guessed from document text is not
    proof the window is open - it is fine for DISPLAY, not for hiding or keeping.
    UNKNOWN is shown, never hidden - a silently dropped tender is a lost bid. Its chip prints the
    source's OWN stage wording (lifecycle.status_text prefers status_details over status) so the
    operator can read "Calificarea ofertantilor" straight off the inbox and add it to
    Company settings -> List behaviour -> results.closed_statuses. That loop is the design: we do
    not hardcode a portal's stage names, the operator teaches them once.
    tender.statusDetails IS now captured (workflows/collectors/mtender.normalize_record). Existing
    rows get it with `app.py renormalize mtender` (POSITIONAL source, there is no --source flag).
    It re-derives normalized_json from raw_documents.payload_json - no re-fetch; the raw package is
    kept forever and only an explicit source reset purges it.
    THE FILTER IS ONLY AS GOOD AS THE STORED RECORD. Tenders collected by an older normalizer keep
    whatever nj they were given - a missing deadline stays missing, so the rule returns UNKNOWN and
    shows them. Changing the normalizer does NOTHING for existing rows until renormalize runs. When
    a filter "does not work", check the DATA first with:
        app.py inspect <ocid-or-id-fragment> [--find <word off the portal>]

KEY FACTS / DECISIONS
=====================================================================================================
- Collection and analysis are SEPARATE: collect only stores status="new"; analysis runs on demand (Analyze page)
  OR automatically after a scheduled collect job if job.analyze is on.
- NEW vs HISTORY vs ARCHIVE is DERIVED, never stored as its own column. The only stored input is tenders.origin
  (incremental|backfill). Daily/scheduled front-page checks = incremental => can be "New"; manual history crawls
  from Sites = backfill => go straight to General/History. This lets testers hammer old data (scope=not_new/
  history) without polluting the fresh New tab. Segment thresholds (results.new_days=7, archive_after_days=90)
  are editable (User Settings / Results).
- Daily automatic collection is a Scheduler on top of app.py serve (or the web process). The schedule is data:
  one {kind:"collect"} entry in schedule.jobs, edited on /schedule. Ships DISABLED — nothing fires until Victor
  turns it on and sets times. Generic-web collection spends tokens; MTender does not (surfaced in the UI).
- The API key lives in ~/tenderengine/.env (engine/secrets), NOT in ConfigStore and NOT in git. Editable via
  the API-key box on User Settings. Missing key => engine silently runs the STUB provider (fake text); the
  User Settings diagnostics shout about this so it isn't mistaken for real analysis.
- PYTHON 3.10 ON THE SERVER: the venv is 3.10.12. Do NOT use 3.11+/3.12-only syntax (e.g. backslash escapes
  inside f-string expressions). run verify_install.py (it py_compiles every file) before restarting so a
  version-only SyntaxError never reaches the live service.
- mtender.md is a JS SPA: its server HTML renders only the first ~7 pages, then JS. Generic scraping caps ~60-70.
  RELIABLE MTender = the sources.mtender OCDS API (confirmed: 1520 collected). Do not scrape mtender.md.
- SharePoint/UNDP doc libraries need login+browser -> analyzer flags needs_login; not scrapeable; use API/RSS.
- Estimate: sites rarely print a total; genericweb derives it from the pagination last_page * per-page count.
- Honest stall: on an empty page below ~90% of estimate, mark NOT exhausted with a note explaining the JS boundary.
- Engines: builtin (plain fetch + optional Playwright) vs crawl4ai (library renders JS + clean markdown). Toggle on /sites.
- Adding a new site no longer requires guessing the JS toggle: "detect & add" (POST /sites/detect) runs a
  cheap-to-expensive funnel on ONE test page (see [6] detect_site_profile) and stores the winning method as
  a per-site config override — no DB migration needed, it's just another field in sites.tenders[] JSON.
- Verified against tenderhub.md/ro (Romanian B2G tender aggregator: UNDP/IOM/UIPAC/UNICEF postings) — it is
  plain server-rendered HTML with a real table, no login for browsing, so builtin/no-render should win the
  funnel there; pagination mechanism (numeric ?page=N vs AJAX) was NOT verified live (sandbox has no network
  route to tenderhub.md) — check this first if incremental/backfill on it behaves oddly.
