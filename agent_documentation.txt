HOW TO WORK
-----------
GIT: repo https://github.com/Madly222/mcp4tender (public). Sandbox can `git clone --depth 1` it
  (github.com/codeload/raw allowlisted). The assistant CANNOT push; Victor applies a delivered tar
  and pushes. A fresh clone at session start is the reliable way to see what's actually in main.
DELIVERY: tar of changed files -> extract into ~/tenderengine, run verify_install.py + the relevant
  tests, `systemctl --user restart tenderengine`, then git add/commit/push (Victor's preference:
  one chained && command so a failure stops the restart and the push).
CODE STYLE: NO comments in code. Small modules, ~300 lines max, split when bigger. Config-driven —
  no hardcoded values that belong in ConfigStore.
TESTS: python -m pytest tests/ -q --ignore=tests/test_phase4b.py (phase4b needs pymupdf; fine on
  the server). verify_install.py py_compiles everything — run before any restart (catches
  3.12-only syntax; the SERVER RUNS PYTHON 3.10: no dt.UTC, no 3.11+ syntax).
SANDBOX: filesystem resets between turns; fastapi/python-multipart/pytest/httpx need
  `pip install ... --break-system-packages`. No network route to real tender sites — mock
  preview/analyze/fetch. Chromium available for file:// renders. ~11 test files assert on raw HTML
  substrings; markup changes break them by design — update assertions, don't delete tests.

GLOBAL CONTRACTS
----------------
- app.py bootstrap()->(conn,store,seeded); web/server.py create_app(db_path). CLI: collect, triage,
  extract, applicability, suppliers, web, serve, user, inspect, renormalize, dedupe-docs, ...
- ConfigStore (engine/config_store.py): get/set(actor,note)/reload/seed_defaults; every set is
  versioned in `configs` (rollback exists in admin /config only — TODO 15). Defaults in
  config/defaults/*.json; seeding adds NEW keys only, never overwrites live values.
- tenders.status: new -> triaged -> extracted -> analyzed -> sourced (failed = stage errored;
  "updated" = source changed a stored tender, re-enters at triage).
- tenders.origin: incremental (scheduled/daily) | backfill (manual history crawl). Segments
  (workflows/segments.py) are DERIVED: new | history | archive from origin+dates; never stored.
- external_id: genericweb "<site_id>:<sha1(url|ref|title)[:16]>"; mtender base-ocid (stage copies
  collapse via base_ocid + analysis.dedupe_mtender).
- DB (engine/db.py): CREATE IF NOT EXISTS + _migrate on boot. Tables: tenders, verdicts,
  extractions, verifications, suppliers, raw_documents, source_state/runs, pipeline_runs,
  stage_runs, stage_events (per-tender step trace), llm_cache, llm_spend, configs, config_meta,
  audit_log, crawl_state, dismissed_tenders, supervisor_runs, accounts, sessions, login_attempts,
  tender_work, qualify_runs, recheck_state.
- LLM: engine/llm.py LLMGateway(store,conn).complete(stage,system,messages,max_tokens,prefill).
  Model per stage from llm.models (haiku default/extract/suppliers/verify; sonnet applicability);
  auto-retry once on a stale model string with llm.models.default. Key via engine/secrets
  (~/tenderengine/.env or env var), NEVER config. Missing key => stub provider; diagnostics shout.

THE LLM CYCLE (worker + controller, confirmed with Victor Jul 23)
-----------------------------------------------------------------
Stage 1 ranking costs NOTHING: triage is keyword/CPV/value scoring, no model. Stage 2 workers:
haiku reads documents (extract), SONNET judges applicability, haiku matches the catalog
(suppliers). Stage 3 controller: workflows/verify.py runs AFTER extract, applicability AND
suppliers, checking the worker's output against the source; strictness=balanced (default since
Jul 23) retries the worker only on MISSING fields. The controller runs on haiku; to make it
"at least as strong as the worker", set llm.models.verify to sonnet in Company settings — check
/app/costs first, it multiplies verify spend. Lifecycle continues after analysis: recheck ([R])
re-reads relevant tenders' pages for status/deadline changes until they close.

HARD-WON DECISIONS — DO NOT UNDO
--------------------------------
- ACHIZITII vs MTENDER (REVERSED Jul 2026, final): same procurement system, two front-ends, but
  achizitii.md is ranked SOURCE #1 for data quality and IS scraped; MTender API stays as
  lower-rank fallback. Do NOT add achizitii to API_COVERED_HOSTS, do NOT remove it from sites.
  The token cost is an accepted trade. Duplicates collapse via cross-source dedup.
- CROSS-SOURCE DEDUP (engine/dedup.py in _store_item): dedup_key = folded title|buyer exact
  match; rank from config sources.rank (lower index = stronger, edited only by the reorder UI);
  a stronger incoming source UPDATEs the stored row IN PLACE (same tenders.id — verdicts/work
  stay attached, status="updated"), a weaker one is dropped; carry_closed keeps awarded/status
  from the stored copy so a scrape can never "reopen" an API-closed tender.
- INBOX = "can we still submit a bid?" (web/user/lifecycle.state_of, most-trustworthy first):
  awarded -> CLOSED; status/status_details in results.closed_statuses -> CLOSED; a known
  submission deadline decides OPEN/CLOSED (outranks any label); pending_statuses -> PENDING;
  else UNKNOWN and SHOWN (chip prints the source's own wording so the operator can teach
  closed_statuses). Never treat "a status exists" as proof bidding is open (OCDS keeps
  status=active through award). Deadline check uses nj["deadline"] only — an LLM-read date from
  docs is display-only, never filter input. MTender: several releases per ocid advance
  independently — status must come from _latest, stable fields from _first. When a filter
  "doesn't work", check the DATA first: app.py inspect <fragment> [--find word]; old rows need
  app.py renormalize mtender (positional arg).
- INBOX IS DECISION-BOXED, NOT TIME-BOXED: results.new_days defaults to 0 = no ceiling; only the
  user (tender_work) or lifecycle-closed removes an undecided tender. A third removal path is a
  bug. Sweep marks the hidden-closed as skipped, reversible.
- NEVER SCRAPE AN API-COVERED HOST (money, missing statuses, and hiding != disabling — the row
  stays visible with a red strip and a Remove button). Host match only.
- TOKEN-LEAK GUARDS (Jul 23): stub responses are NEVER written to llm_cache and a poisoned
  "STUB_RESPONSE" cache entry is purged on read by a real provider. verify.strictness default
  balanced (strict re-ran worker+controller on any flag — on applicability that's an extra
  sonnet). Recheck hashes text not raw HTML and skips dead tenders. Incremental page-1 hash gate.
  Analyze-now pings via health.check_api_key, not the cache-answered gw.complete.
- EMAIL: Date + Message-ID always (UBE rejection was live). Multi-recipient = comma list in
  notify.email.to.
- MULTI-COMPANY IS SOLVED BY DEPLOYMENT, NOT SCHEMA: one instance per company = data dir + .env +
  systemd unit + port, ONE shared checkout (env TENDERENGINE_DB / TENDERENGINE_HOME; port from
  that instance's store). Do NOT add company_id to tenders/verdicts/suppliers/extractions; the
  "3b multi-tenancy" plan is CANCELLED. Scheduler thread owns its own store — ambient-config
  swapping is exactly the leak source this avoids.
- SETTINGS UX: no raw JSON, no machine names, every generated key labeled (tests enforce),
  unticked checkbox = OFF (forms write ALL booleans), currency is a <select> because a typo'd
  currency silently zeroed line costs. Preferences = per-person view knobs (theme) ONLY;
  everything that changes tenders/cost is Company settings.
- QUALIFIED SORT: deadline default with N/A last via day_end_ts; the old unconditional re-sort
  inside the handler was removed — the ?sort= param is the single authority.
- web.cookie_secure stays False until TLS fronts the app; flipping early kills sessions over HTTP.
